## Q9：独立 Embedding 模型是怎么训练的？和 LLM 的关系？

### 先厘清概念

LLM 里有两种"embedding"，**不是一回事**：

| 类型 | 位置 | 作用 | 维度 | 
|------|------|------|------|
| **内部 Token Embedding** | LLM 模型的第一层 | 把 token ID → 向量，让 Transformer 处理 | 和 hidden_size 相同 |
| **独立 Embedding 模型** | 独立于 LLM 的专用模型 | 把任意文本 → 向量，用于检索/聚类/语义搜索 | 通常不同（可配置） |

### 独立 Embedding 模型怎么训练

独立 Embedding 模型（如 OpenAI text-embedding-3-small, BGE, E5, Cohere embed）的训练方式**和 LLM 完全不同**：

#### 方法一：对比学习（Contrastive Learning）——最主流

核心思想：让语义相似的文本向量距离近，不相似的远。

```
训练数据格式：
{
  "query": "如何训练神经网络",
  "positive": "神经网络训练方法包括反向传播...",  ← 相关文档
  "negative": "今天天气真好"                       ← 不相关文档
}
```

训练目标 = **InfoNCE Loss**：
```
                    exp(sim(query, positive) / τ)
loss = -log ──────────────────────────────────────────
            exp(sim(query, positive)/τ) + Σexp(sim(query, negative_i)/τ)
```

直观理解：query 和 positive 做点积（cosine sim），和所有 negative 也做点积。让正例的点积大、负例的点积小。这个目标叫 **把正例从负例中"拽"出来**。

#### 方法二：RetroMAE / SimCSE——无监督

只用单条文本，不做配对：
- 给文本加噪声（随机 mask 部分 token）
- 让模型的 embedding 能还原被 mask 掉的 token
- 效果：模型必须理解整段语义才能完成这个任务，从而学到了好的语义向量

#### 方法三：基于 LLM 蒸馏

用 GPT-4/Claude 等大模型生成海量 `(query, relevant_doc)` 对，然后训练小 embedding 模型拟合这些关系。OpenAI 的 text-embedding-3 就是这条路。

### 和 LLM 的关系

```
LLM 的解码过程：
token IDs → [Token Embedding] → [Transformer×N] → [lm_head] → 下一个 token

Embedding 模型的编码过程：
token IDs → [Token Embedding] → [Transformer×N] → [pooling] → 一个向量
```

**关键区别**：

| 维度 | LLM | Embedding 模型 |
|------|-----|---------------|
| **输出** | 序列中每个位置预测下一个 token | 整段文本的**一个固定长度向量** |
| **Attention** | 因果（Causal），只看左边 | 双向（Bidirectional），看全部上下文 |
| **训练目标** | 下一个词预测 | 对比学习 / 相似度学习 |
| **最后一层** | lm_head（hidden→vocab） | Pooling（取平均或取 [CLS]） |
| **参数量** | 通常很大（7B-数百B） | 通常较小（100M-7B） |

**相通之处**：
- 底层都是 Transformer
- 都从 Token Embedding 开始
- 很多 Embedding 模型直接**基于 LLM 的 backbone 改造**（拿掉 lm_head，加上 pooling 层，换训练目标）

### 实际案例

**BGE (BAAI General Embedding)**：
```
基座模型：BERT / LLaMA（去掉 lm_head）
加上：Pooling 层（mean pooling）
训练：对比学习（Contrastive Learning）
输出：一个 1024 维向量
用途：语义搜索、RAG 检索
```

**E5 (EmbEddings from bidirEctional EncOder representatiOns)**：
```
基座模型：改造的 LLaMA（从 causal 改为 bidirectional attention）
训练：对比学习（海量 text pairs from web）
输出：一个向量
用途：通用语义匹配
```

### 实战中的配合使用

```
用户提问 → Embedding 模型 → 向量 → 向量数据库（找相似文档）
                                                  ↓
找到的文档 → 拼入 prompt → LLM → 最终回答
```

这就是 **RAG（检索增强生成）** 的基础流程：Embedding 模型负责"大海捞针"，LLM 负责"阅读并总结"。

## Q10：Embedding 占比合理值是多少？

合理范围：**5%-15%**

### 计算

```python
embedding_params = vocab_size × hidden_size
total_params = embedding_params + Σ(每层Transformer参数) + lm_head
ratio = embedding_params / total_params
```

### 几个实际案例

| 模型 | vocab_size | hidden_size | Embedding 参数 | 总参数 | 占比 |
|------|-----------|-------------|---------------|--------|------|
| **GPT-2 Small** | 50,257 | 768 | 38.6M | 124M | **31%** 🔴 偏高 |
| **MiniMind** | 6,400 | 768 | 4.9M | 64M | **7.7%** 🟢 合理 |
| **LLaMA 7B** | 32,000 | 4,096 | 131M | 6.7B | **1.96%** 🟢 |
| **LLaMA 70B** | 32,000 | 8,192 | 262M | 68.9B | **0.38%** 🟢 |
| **DeepSeek-V2** | 102,400 | 4,096 | 420M | 236B (MoE) | **0.18%** 🟢 |

### 规律

**模型越小，embedding 占比越大**。小模型（<1B）需要特别关注 embedding 占比，大模型（>7B）基本不用管。

- **偏低（<1%）**：单词表相对于模型已经太小，可能丢语义信息
- **合理（1%-15%）**：健康范围
- **偏高（>20%）**：大量参数浪费在词嵌入上，Transformer 层不够深

### GPT-2 Small 31% 是一个历史设计失误

GPT-2 的 vocab_size=50,257 对 124M 模型来说太浪费了——近 1/3 的参数卡在 Embedding 层，Transformer 只有 12 层。后续模型（LLaMA 等）都吸取教训，用 32k 词表配大模型。

## Q11：词表越大，常见组合越可能成为单个 token？

**完全正确！** 这就是词表大小的核心权衡。

### 演示

```
语料中出现频率统计：
"比" + "如" 相邻出现 10000 次
"天" + "气" 相邻出现 50000 次  
"人工" + "智能" 相邻出现 30000 次

小型词表（V=1000）：
最多只能保留 1000 个 token
→ "天气" 被合并，"比如" 也可能被合并
但"人工智能" 可能就太长了 → 拆成 "人工" + "智能"

中型词表（V=6400）：
"天气" ✅  "比如" ✅  "人工" ✅  "智能" ✅
"人工智能" → 2 个 token

大型词表（V=100k）：
"天气" ✅  "比如" ✅  "人工智能" ✅（直接是 1 个 token）
甚至 "机器学习" ✅ "深度学习" ✅
```

### 词表大小的核心权衡

| 方向 | 好处 | 坏处 |
|------|------|------|
| **大词表** | 更多常见词/短语整体保留→序列长度短、信息密度高 | Embedding 层大→参数多→显存高；罕见 token 多→训练效率低 |
| **小词表** | Embedding 层小→省参数；但 | 每个 token 信息量少→序列长→训练慢 |

**但有一个关键 nuance**：词表大不一定好，因为词表的增长不是线性的：
- 前 10k 个 token：覆盖 95%+ 的常见词
- 后 90k 个 token：只覆盖 4.99% 的边缘词
- 最后面几千个 token：几乎用不到

所以主流模型多选择 **32k-128k** 这个区间。

### 举例：token 数量对比

同一个句子 "大型语言模型的训练需要大量计算资源"，不同词表的 token 数量：

| 词表大小 | token 数量 | 说明 |
|---------|-----------|------|
| 6400（minimind） | ~10 个 | 常见组合(训练/模型/计算)一个 token，低频拆成单字 |
| 32000（LLaMA） | ~7 个 | 更多短语可完整保留 |
| 100k+（GPT-4） | ~5 个 | "计算资源"甚至可能一个 token |

### 还有其他影响因素

除了词表大小，还有：
1. **BPE merge 次数**：同词表大小下，训练数据的质量和分布不同，结果也不同
2. **数字 tokenization**：GPT-2 把每个数字单独拆分（"2048"→"2""0""4""8"），LLaMA 能把数字组合保留
3. **空格处理**：有的保留前导空格（GPT-2），有的不保留

## Q12：minimind `ceil(hidden×π/64)×64` 公式含义 + hidden_size 到底怎么定？

### 这个公式做了什么

```python
intermediate_size = math.ceil(hidden_size * math.pi / 64) * 64
```

逐层拆解：
- `hidden_size × π`：乘以 π，让中间层大小带一个无理数系数
- `/64`：除以 64 再 `ceil`：让结果是 64 的倍数（GPU 友好）
- `× 64`：恢复倍数

具体值：
```
hidden_size=768 → ceil(768×3.14159/64) × 64
               = ceil(37.699) × 64
               = 38 × 64
               = 2432
```

标准 FFN 经验公式本是 `intermediate_size ≈ 4 × hidden_size = 3072`。2432 略小于 4×，相当于约 **3.17×**。

**这个公式不是标准做法**，更像是 minimind 作者的一个小趣味（π 是个浪漫的数字），其真正意图是：
1. 让中间层是 64 的倍数（GPU 计算对齐）
2. 比标准 4× 略小一点，适合小模型

### 那 hidden_size 到底怎么定的？

这是整个 LLM 架构中的**顶层设计决策**，没有唯一公式，但有**经验法则**：

#### 第一步：决定总参数量

```
你有多大的算力/显存？
→ 目标总参数量
```

#### 第二步：决定层数

```
经验法则：
小模型 (64M-1B):   8-24 层
中型模型 (1B-10B):  32-48 层
大型模型 (10B+):    48-96 层
```

#### 第三步：通过参数量公式反推 hidden_size

```python
# 简化版参数计算公式：
# 每层 Transformer ≈ 4×hidden_size² + attention 投影
# Embedding = vocab × hidden_size
# LM Head ≈ vocab × hidden_size (如果有权重绑定则不重复)
# 总参数 ≈ N_layers × 12 × hidden_size² + 2 × vocab × hidden_size

# 以 minimind 为例：
# 目标 64M，层数=8，vocab=6400
# 8 × 12 × h² + 2 × 6400 × h ≈ 64,000,000
# → h² ≈ (64M - 6400h) / 96
# → h ≈ 768 ✅
```

#### 第四步：硬件对齐

- hidden_size 最好是 64 或 128 的倍数（GPU Tensor Core 对齐）
- head_dim = hidden_size / n_heads 最好是偶数（RoPE 需要）
- minimind: 768 → 8 heads → head_dim=96 ✅

#### 经验速查表

| 目标参数量 | 推荐 hidden_size | 推荐层数 |
|-----------|-----------------|---------|
| ~60M | 768 | 8 |
| ~350M | 1024 | 16 |
| ~1B | 2048 | 16 |
| ~7B | 4096 | 32 |
| ~70B | 8192 | 80 |
| ~200B | 12288 | 96 |

## Q13：绝对位置编码是怎么做的？和 Q/K 有关联吗？

### GPT-2 的绝对位置编码

看 LLMs-from-scratch ch02 的实现：

```python
# Token embedding
token_embedding = nn.Embedding(vocab_size, d_model)
# 位置 embedding
position_embedding = nn.Embedding(max_len, d_model)  # ← 又一个 Embedding！

# 使用：直接相加
input = token_embedding(token_ids) + position_embedding(position_ids)
```

**关键**：位置 embedding 是一个**可学习的参数表**，形状是 `[max_seq_len, hidden_size]`。位置 0 有自己的一行向量，位置 1 有另一行……都是**训练中学出来的**。

### 和 Q/K 的关系——RoPE vs 绝对位置

这恰恰是两者最核心的差异：

| | **绝对位置编码（GPT-2）** | **RoPE（minimind/LLaMA）** |
|--|--------------------------|--------------------------|
| **在哪加** | 在**输入端**，加到 token embedding 上 | 在 **Attention 内部**，旋转 Q/K |
| **和 Q/K 的关系** | **无关**——位置是 token 的附赠信息，Q/K 本身不"知道"位置 | **直接编码进 Q/K**——Q/K 自身就包含位置信息 |
| **数学** | `x' = embed(token) + embed(pos)` | `Q' = Rotate(Q, pos)` |
| **可扩展性** | ❌ 超出 max_len 就不能用 | ✅ 可通过 YaRN 扩展 |
| **相对位置感知** | ❌ 没有（要靠注意力学出来） | ✅ 天然有（Q(m)·K(n)∝cos(m-n)） |

### 绝对位置的局限

这就是为什么新模型都转向 RoPE：

```
绝对位置：见过位置 0-2047 的向量
位置 2048 → 没有对应向量 → 需要重新训练

RoPE：见过周期 0-2047 的旋转角度
位置 2048 → 继续旋转 2048° × ω
         → 超出训练范围但依然有数学定义
         → 可以通过 NTK/YaRN 微调继续用
```

### GPT-2 位置 Embedding 的可视化

如果把训练好的位置 embedding 做 PCA 降到 2D，会发现：
```
位置 0━━━━━━━━━━━━━━━位置 10━━━━━━━━━━位置 50━━━━━━
  连续变化，相邻位置向量相近，远距离位置向量不相关
```
这说明位置 Embedding 学到了"位置相似性"的模糊概念，但不如 RoPE 的数学性质精确。

## Q14：NTK-aware 维度频率是什么？

### 先理解 RoPE 的频率分布

RoPE 的核心公式 `freqs = 1 / θ^{2i/d}`：

```python
theta = 10000.0
d = 64  # head_dim

维度 0:  freq = 1 / 10000^{0/64} = 1/1 = 1
维度 1:  freq = 1 / 10000^{2/64} = 1/10^{0.5} ≈ 0.316
维度 2:  freq = 1 / 10000^{4/64} = 1/10 ≈ 0.1
...
维度 31: freq = 1 / 10000^{62/64} = 1/10^{31/32} ≈ 0.107
维度 32: freq = 1 / 10000^{64/64} = 1/10000 = 0.0001
```

看两个极端：

**高频维度（维度 0）**：
```
freq = 1 → 每 1 个 token 位置就完成一整圈旋转
比如位置 0: 角度 0° → 位置 1: 角度 360° → 位置 2: 角度 720°
这个维度对位置极其敏感——"紧挨着"的两个 token 都能区分
```

**低频维度（维度 32）**：
```
freq = 0.0001 → 每 62832 个 token 才完成一整圈
位置 0: 角度 0° → 位置 1000: 角度 0.1° ≈ 几乎没动
这个维度只能区分"很远"的位置——"位置 3 和位置 5"它分不清
```

### 频率分布特性

```
频率值（越高越敏感）：
1.000 ┤█
0.316 ┤██
0.100 ┤ ██
0.032 ┤   ██
0.010 ┤     ██
0.003 ┤       ██
0.001 ┤         ██
0.000 ┤           ████████████
       └──► 维度编号
       高频端←──────────→低频端
       (敏感，区分近邻)   (迟钝，只区分远距)
```

前几个维度频率高→位置分辨力强
后几个维度频率低→位置分辨力弱

### NTK-aware 的核心理念

NTK-aware 利用了**不同维度敏感度不同**的特性：

**对高频维度（敏感）**：几乎不压缩
```
原来：维度 0 在 0-2048 范围内转了 2048 圈
扩展后：让它仍然在 0-8192 范围内转 2048 圈 ← 不缩
效果：附近位置依然能清晰区分
```

**对低频维度（迟钝）**：大幅压缩
```
原来：维度 32 在 0-2048 内只转了 0.2 圈
扩展后：让它转到 0-8192 也只转 0.2 圈 ← 缩 4 倍
效果：本来就分不清近邻，缩了也不影响区分远距离
```

### 为什么叫"NTK"（Neural Tangent Kernel）？

这个名字借用了神经网络理论中的一个概念：NTK 表明模型不同层的神经元对不同频率的输入敏感度不同。RoPE 的维度频率也呈现类似的谱分布，所以采用了这个命名。

### 直观理解

```
扩展前（训练 2048）：
视角：我有一台望远镜，能看清远处 2048 米内的所有细节
高频维度：负责"相邻 1 米"的分辨（敏感）
低频维度：负责"近 1000 米和远 2000 米"的分辨（迟钝）

扩展后（推理 8192）：
视角：望远镜要看清 8192 米
选项 A（PI）：
  - 把所有刻度均匀拉长 → 相邻 1 米变成了相邻 4 米 → 近处细节丢失
选项 B（NTK/YaRN）：
  - "相邻 1 米"的高频镜头不缩放 → 近处细节保留！
  - "远距离"的低频镜头缩放 4 倍 → 反正它本来也看不清近处
  - 人到中年学到的事实：用对的人做对的事
```

### NTK 在 minimind 中的实现

```python
# 计算高低频过渡区
high = ceil(inv_dim(beta_slow, dim, rope_base))  # ~32
low = floor(inv_dim(beta_fast, dim, rope_base))  # ~1

# ramp 从 0→1：维度 < low → 不缩；维度 > high → 全缩；中间线性过渡
ramp = clamp((arange - low) / (high - low), 0, 1)

# 频率调整
freqs = freqs × (1 - ramp + ramp / factor)
#         ↑ 高频(ramp≈0)不变      ↑ 低频(ramp≈1)缩 factor 倍
```

这个过渡区保证了高频和低频之间的平滑衔接，不会在某个维度上突然断裂。
