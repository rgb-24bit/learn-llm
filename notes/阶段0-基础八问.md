## Q1：full_sft → lora → dpo → ppo → grpo → agent_rl → 蒸馏 各自是啥？

这些全部属于**后训练（Post-training）**阶段，是在预训练好的基座模型上做的一系列"调教"操作。

### 按"调教深度"排列：

```
预训练（基座模型）
  │
  ├─ SFT（Full SFT）：全参数指令微调，用对话数据教会模型"问啥答啥"
  ├─ LoRA：参数高效微调，只改一小部分参数，效果接近全量微调
  ├─ DPO：直接偏好优化，让模型学会"选好的、拒绝坏的"
  ├─ PPO：经典强化学习对齐，需要一个 Reward Model 打分
  ├─ GRPO：Group-based PPO，DeepSeek-R1 用的，不需要 Reward Model
  ├─ Agentic RL：在工具调用场景下的 GRPO/CISPO，让模型学会"先查天气再回答"
  └─ Distillation：蒸馏，大模型的知识压缩到小模型
```

### 逐一详解：

#### SFT（Supervised Fine-Tuning）—— 指令微调
- **做什么**：给模型看大量`{"instruction": "...", "output": "..."}` 对话对
- **本质**：监督学习，让模型的输出模式从"续写文本"变成"回答问题"
- **minimind 实现**：`train_full_sft.py`，和 pretrain 几乎一样，只是数据集换成了对话格式
- **特点**：全参数更新，效果最好但最贵

#### LoRA（Low-Rank Adaptation）
- **做什么**：不更新原模型的 768×768 矩阵，而是旁边挂一个 `768×r + r×768` 的小矩阵（r=8 或 16）
- **r 很小**，所以参数量只有原来的 1/50 左右
- **本质**：`W_new = W_old + A×B`，只训练 AB，W_old 冻结
- **minimind 实现**：`model/model_lora.py` + `train_lora.py`，从零实现不依赖 peft
- **为什么有效**：研究发现大模型的参数变化集中在低秩子空间

#### DPO（Direct Preference Optimization）
- **做什么**：给模型看"好回答 vs 坏回答"的对子，直接优化偏好
- **关键**：不需要单独的 Reward Model，只用同一个模型的输出来对比
- **loss 公式简化理解**：让模型给"好回答"的得分 > "坏回答"的得分，且差距越大越好
- **minimind 实现**：`train_dpo.py`，核心就几行：
  ```python
  pi_logratios = chosen_probs - rejected_probs
  ref_logratios = chosen_ref_probs - rejected_ref_probs
  logits = pi_logratios - ref_logratios
  loss = -logsigmoid(beta * logits)
  ```

#### PPO（Proximal Policy Optimization）
- **做什么**：经典 RLHF 流程，需要一个训练好的 Reward Model 给回答打分
- **流程**：生成回答 → Reward Model 打分 → 用 PPO 更新模型，让高分回答的概率变大
- **复杂程度**：最高，需要同时维护 4 个模型（Actor, Reference, Reward, Critic）
- **minimind 实现**：`train_ppo.py`

#### GRPO（Group Relative Policy Optimization）
- **做什么**：DeepSeek-R1 用的方法，PPO 的简化版
- **区别**：不需要 Reward Model，对同一个 prompt 生成多个回答，内部互相比较
  ```
  prompt: "1+1=?"
  生成 8 个回答: [2, 2, 2, 3, 2, 1, 2, 4]  ← 5/8 对了
  奖励 = (这个回答的对错) - (组的平均正确率)
  对的且比平均好的 → 加分 | 错的 → 减分
  ```
- **minimind 实现**：`train_grpo.py`，核心代码 `calculate_rewards` 函数

#### Agentic RL
- **做什么**：让模型学会**在工具调用场景下**做强化学习
- **场景**：用户问"北京天气怎么样？" → 模型应该先调 `get_current_weather("北京")` → 拿到结果再回答
- **minimind 实现**：`train_agent.py`，定义了 6 个工具（天气、汇率、翻译等），用 GRPO/CISPO 训练模型正确使用工具
- **数据**：`agent_rl.jsonl` 和 `agent_rl_math.jsonl`

#### Distillation（模型蒸馏）
- **做什么**：用一个大模型（Teacher）的输出做软标签，训练小模型（Student）
- **本质**：大模型说"这道题选 A 的概率 0.7，B 是 0.2" → 小模型模仿这个分布，而不是硬学"答案是 A"
- **minimind 实现**：`train_distillation.py`

### 各阶段的关系图示

```
Pretrain（基座模型——会说但不会回答问题）
    ↓
Full SFT（学会对话格式——会问啥答啥了）
    ├── LoRA（节省资源，效果逼近全量）
    ├── DPO（偏好对齐——学会说好话）
    ├── PPO/GRPO（强化学习对齐——更稳定、更前沿）
    ├── Agentic RL（学会用工具——能联网查询了）
    └── Distillation（压缩到小模型——省钱省算力）
```

## Q2：预训练是让知识嵌入模型，后训练是让模型学会如何使用这些知识？

**对，这个理解基本正确，但可以更精确一点。**

### 预训练（Pretrain）
- **输入**：海量互联网原始文本（TB 级）
- **任务**：「下一个词预测」
- **学到的东西**：
  - 语法知识（"我吃饭"比"吃饭我"更合理）
  - 世界知识（"北京是中国的首都"）
  - 推理能力（"如果 A > B 且 B > C，那么 A > C"）
  - 上下文关联能力
- **关键**：这些知识是**隐式编码**在数十亿参数中的，不是像数据库一样存储

### 后训练（Post-training）
- **输入**：少量高质量数据（MB 级）
- **做的**：**调整模型输出分布**，相当于给训练好的神经网络上加一个"行为过滤器"
- **学到的东西**：
  - 格式控制（用户问 → 模型答的对话格式）
  - 偏好对齐（什么回答算好、什么算差）
  - 安全性（拒绝恶意请求）
  - 角色定位（"你是一个有用的 AI 助手"）

### 比喻理解

```
预训练 = 把一个人扔进图书馆，让他自己读完所有书
         他学到了大量知识，但不知道怎么用

SFT    = 教他：当别人问问题时要回答，而不是继续自言自语
DPO    = 教他：回答要礼貌、准确、有用，不要胡说八道
GRPO   = 让他在实践中自我纠错，答错了自己反思
Agent  = 教他：不知道答案时可以查资料（调用工具）
```

### 但要注意一个 nuance

预训练**不只是**"存储知识"，它也在学推理模式。有研究表明，即使没有 SFT，预训练模型也有一定的问答能力——只是格式不对、回答质量差。后训练是在**做格式对齐和偏好筛选**，而不是凭空"教会"模型使用知识。

## Q3：Tokenizer 词表大小如何确定？逐段讲解训练代码

### 词表大小怎么定？

几个关键因素：

| 因素 | 说明 | 典型值 |
|------|------|--------|
| **语言** | 中文 token 密度远低于英文（一个中文字就是一个 token，英文一个词拆成多个 token） | 中文模型通常更大 |
| **模型规模** | 小模型用小词表（避免 Embedding 层太大） | minimind: 6,400 |
| **压缩率** | 词表越大，每个 token 携带的信息越多，但稀有 token 也越多 | GPT-4: ~100k |
| **训练效率** | 词表越大 → Softmax 输出层越贵 | 速度 vs 质量的权衡 |
| **tokenizer 保存** | 词表在预训练之前就定死，不会再改 | 一旦确定用到模型退役 |

**经验法则**：
- 50k 以下：轻量模型/英文（GPT-2: 50,257）
- 50k-100k：主流 LLM（LLaMA: 32k → 128k 各版本不同）
- 100k+：需要覆盖多语言的模型

minimind 选 6400 是因为它本来就是个极轻量的教学模型，Embedding 层 `6400×768 ≈ 4.9M` 参数，对总参数量 64M 来说是合理比例。GPT-2 的 `50257×768 ≈ 38.6M` 占比就高很多了。

### 逐段讲解训练代码

用 yield 做成生成器，不一次加载全部文本到内存。

## Q4：Token Embedding 向量维度如何确定的？最佳实践？

### 怎么定下来的？

**不是拍脑袋定的，而是整个模型架构的参数权衡。** 看 minimind 的配置：

```python
MiniMindConfig(
    vocab_size=6400,          # 词表大小
    hidden_size=768,          # ← 这就是 embedding 维度！
    num_hidden_layers=8,      # Transformer 层数
    num_attention_heads=8,    # 注意力头数
    head_dim=96,              # 每个头的维度 = hidden_size/num_heads? 不!
    # head_dim = 96，而 768/8=96，所以 head_dim = hidden_size/num_heads
)
```

所以 `hidden_size = 768` 就是 embedding 向量的维度，也是每层 Transformer 的隐藏层宽度。

### 维度选择的核心约束

```
Embedding 参数量 = vocab_size × hidden_size
                  = 6400 × 768 ≈ 4.9M

总参数量 ≈ 8×(768²×4 + ...) ≈ 64M
Embedding 占比 ≈ 4.9/64 ≈ 7.7%    ← 合理的比例
```

如果 hidden_size 太大会怎样？
| hidden_size | Embedding 参数 | 问题 |
|-------------|---------------|------|
| 768 | 4.9M | ✅ 合理 |
| 2048 | 13.1M | 对小模型来说占比过高 |
| 4096 | 26.2M | 总参量爆炸 |

### 最佳实践

**有一个经典的"经验公式"：**

```
d_model : d_ff : num_heads : vocab_size
≈ 1 : 4 : d_model/64 : 权衡
```

| 模型 | hidden_size | 层数 | 参数量 |
|------|-------------|------|--------|
| GPT-2 Small | 768 | 12 | 124M |
| GPT-2 XL | 1600 | 48 | 1.5B |
| LLaMA 7B | 4096 | 32 | 6.7B |
| LLaMA 70B | 8192 | 80 | 68.9B |
| MiniMind | 768 | 8 | 64M ← 教学级 |

**趋势：** 模型越大 → hidden_size 与层数一起增加。DeepSeek 等新模型更偏好更大的 `d_ff`（FFN 中间层宽度），minimind 用了 `intermediate_size = ceil(hidden_size × π / 64) × 64`——没错它把 **π** 写了进去！

## Q5：输出和输入权重绑定是为什么？是主流吗？

### 为什么做权重绑定

看 minimind 的代码：

```python
class MiniMindForCausalLM(PreTrainedModel):
    def __init__(self, config):
        self.model = MiniMindModel(config)         # 含 embed_tokens（输入）
        self.lm_head = nn.Linear(hidden_size, vocab_size)  # 输出层
        if config.tie_word_embeddings:              # 如果启用绑定
            self.model.embed_tokens.weight = self.lm_head.weight  # ← 同一份权重！
```

这意味着：
- **输入层**：`Embedding[vocab_size, hidden_size]` 把 token ID → 向量
- **输出层**：`Linear[hidden_size, vocab_size]` 把向量 → 词表概率
- **绑定**：它们用的是 **同一套参数**

### 为什么有效？

因为它们在逻辑上是**互逆操作**：
```
输入：token_ID_123 → embed → [0.1, -0.3, ...]   # 编码
输出：[0.1, -0.3, ...] → lm_head → 概率分布   # 解码
```
它们本质上在做相同的事情——建立"token 和向量"之间的映射关系。共享权重是合理的正则化约束。

### 好处
1. **省参数量**：节省 vocab_size × hidden_size 个参数（对 GPT-2 是 38.6M，约 30%）
2. **正则化**：让正反向映射一致，提升泛化性
3. **加速收敛**：训练初期就有一个合理的对称性

### 是不是主流？

| 阵营 | 做法 | 代表模型 |
|------|------|---------|
| ✅ **绑定派** | tie_word_embeddings=True | GPT-2, BERT, MiniMind, 很多千亿参数以下模型 |
| ❌ **不绑定派** | 独立 lm_head | LLaMA, Qwen, DeepSeek（近年主流） |

**趋势**：新模型（LLaMA 系列、Qwen、DeepSeek）**大多不绑定了**。原因：
1. 大模型 vocab_size 大（100k+），Embedding 层已经是巨无霸，不在乎输出层那点
2. 独立参数让输出层有更多灵活性
3. 现代训练有更好的正则化手段，不需要靠绑定做约束

**minimind 这里默认绑定**是因为它只有 64M 参数，绑定能省大约 6400×768 ≈ 4.9M 参数（约 8%），积少成多。

## Q6：RoPE 的 Q/K 值怎么来的？Q/K 不是 Attention 里的内容吗？

**你的直觉是对的——Q/K 确实是 Attention 的核心。RoPE 是在 Q/K 上动手脚，而不是独立于 Attention 的东西。**

### 标准 Attention 中的 Q/K

在没有位置编码时：
```
输入: [词0的向量, 词1的向量, 词2的向量]
    → Q = 词向量 × W_Q
    → K = 词向量 × W_K
    → Attention Score = Q × K^T   ← 这里！QK 是点积
```

问题是：词0和词1点积的结果，跟词0在位置0、词1在位置1**没有关系**——如果交换它们的位置，点积结果不变。

### RoPE 插入的位置

看 minimind 的 Attention 代码：

```python
def forward(self, x):
    xq = self.q_proj(x)   # Q: [batch, seq_len, heads, head_dim]    ← 标准 Q
    xk = self.k_proj(x)   # K: [batch, seq_len, heads, head_dim]    ← 标准 K

    # ⬇️ RoPE 插入在这里！⬇️
    xq, xk = apply_rotary_pos_emb(xq, xk, cos, sin)
    # xq 和 xk 被旋转了，然后才是后面的 Attention 计算

    # 标准 Attention（rotation-aware 版本）
    scores = xq @ xk.T / sqrt(head_dim)
```

**所以流程是：**
```
词向量 → 投影得到 Q/K → 用 RoPE 旋转 Q/K → Attention 计算
                              ↑
                        这是新加的
```

### RoPE 做了什么旋转？

```python
def apply_rotary_pos_emb(q, k, cos, sin):
    def rotate_half(x):
        # 把 x 的后半部分取负，交换到前半部分
        return torch.cat([-x[..., dim//2:], x[..., :dim//2]], dim=-1)
    
    q_embed = q * cos + rotate_half(q) * sin
    k_embed = k * cos + rotate_half(k) * sin
    return q_embed, k_embed
```

**物理含义**：Q 和 K 的每一对维度 `(d, d+1)` 组成一个二维平面，在这个平面上旋转一个角度 = 位置编号 × 频率。

```
位置 0 的 q:  (q[0], q[1]) → 旋转 0°
位置 1 的 q:  (q[0], q[1]) → 旋转 1° × ω
位置 2 的 q:  (q[0], q[1]) → 旋转 2° × ω

位置 0 的 k:  (k[0], k[1]) → 旋转 0°
位置 1 的 k:  (k[0], k[1]) → 旋转 1° × ω
```

### RoPE 点积的数学性质

当 Q 旋转了 m°，K 旋转了 n° 时：
```
Q(m) · K(n) = f(Q, K) × cos((m-n) × ω)
```

**关键发现**：点积结果只取决于 (m-n)——即相对位置！距离为 3 的两个 token，不管在开头还是结尾，Attention 的"位置影响"是一样的。

这就是 RoPE 的精髓：**把位置信息编码进 Q/K，让 Attention 天然感知相对位置，且没有额外参数需要学习。**

## Q7：推理时上下文窗口大小扩展的原理

### 根本问题

训练时模型只见过固定长度的序列（比如 2048），推理时如果想处理更长的上下文（比如 16k），RoPE 中的 `t = torch.arange(end)` 会取到训练时从未见过的数值。

简单扩充到长序列后，那些从未训练的"高频率"会被推到超出正常范围，导致模型输出质量下降。

### 三种扩展方法（从简单到复杂）

#### 1️⃣ 位置插值（Position Interpolation, PI）
```
原来位置 0-2048
现在要扩展到 0-8192

做法：把新位置缩回到原范围
位置 8192 → 当作位置 2048 来算
位置 4096 → 当作位置 1024 来算
```
**问题**：高频维度压缩严重，模型可能分不清细微位置差异。

#### 2️⃣ NTK-aware 插值
RoPE 中每个维度有不同的"频率"：
```
维度 0:  高频（每 1 个位置就转一整圈），敏感
维度 64: 低频（每 65536 个位置才转一圈），很迟钝
```
NTK 的做法：**高频率少压缩，低频率多压缩**
- 高频维度：几乎不用缩放（因为原来就很密集）
- 低频维度：大幅缩放（反正它们也分辨不了细节）

这样既保持了对局部位置的敏感度，又能覆盖更长的范围。

#### 3️⃣ YaRN（Yet another RoPE extensioN method）
在 NTK 基础上进一步优化——下一题详讲。

## Q8：YaRN 是什么

### 全称
**YaRN** = Yet another RoPE extensioN method
论文：YaRN: Efficient Context Window Extension of Large Language Models

### 核心思路

YaRN 其实做了**两件事**：

#### 事一：频率调整（类似 NTK）
```python
def precompute_freqs_cis(dim, end, rope_base, rope_scaling):
    if rope_scaling is not None:
        orig_max = rope_scaling["original_max_position_embeddings"]  # 2048
        factor = rope_scaling["factor"]                               # 16
        beta_fast = rope_scaling["beta_fast"]                         # 32
        beta_slow = rope_scaling["beta_slow"]                         # 1
        
        # 计算"临界维度"：高于此维度的频率不做调整
        inv_dim = (dim * log(orig_max / (b * 2π))) / (2 * log(rope_base))
        low = max(floor(inv_dim(beta_fast)), 0)
        high = min(ceil(inv_dim(beta_slow)), dim//2 - 1)
        
        # 构建 ramp：从 low 到 high 线性过渡
        ramp = clamp((arange(dim//2) - low) / (high - low), 0, 1)
        
        # 调整频率：高频（维度 0）不调整，低频维度大调整
        freqs = freqs * (1 - ramp + ramp / factor)
```

**示意图：**
```
频率缩放因子
1.0 ┤███████████████████████████████░░░░░░░░░░░───
    │  维度 0 (高频)       ↑           维度 64 (低频)
    │                 临界点
    │                 低 → 高过渡区
1/16 ┤───────────────────────────────────────────
```
高频维度（靠近 0）：不缩放，保留位置敏感度
低频维度（靠近 64）：缩放 1/16，覆盖更长距离
中间维度：线性过渡

#### 事二：Attention logits 缩放

长上下文时注意力分布会变得更平坦（softmax 后更均匀），影响模型对关键位置的关注。YaRN 引入 `attention_factor` 来缩放注意力权重，补偿这个效应。

```python
freqs_cos = cos(freqs) * attn_factor
freqs_sin = sin(freqs) * attn_factor
```

### YaRN vs PI vs NTK 对比

| 方法 | 思路 | 效果 |
|------|------|------|
| PI | 均匀缩放所有位置 | 简单但近邻位置分辨率下降 |
| NTK | 高频少缩、低频多缩 | 效果好于 PI |
| **YaRN** | NTK + attention 缩放 | **目前最佳实践** |

### minimind 中的 YaRN 配置

```python
rope_scaling = {
    "type": "yarn",              # 启用 YaRN
    "factor": 16,                 # 扩展 16 倍
    "original_max_position_embeddings": 2048,  # 原训练长度
    "beta_fast": 32,              # 高频过渡上界
    "beta_slow": 1,               # 低频过渡下界
    "attention_factor": 1.0       # attention 缩放（可调）
}
```

这意味着：本来只能在 2048 长度下训练和推理的模型，可以在 2048×16=32768 长度的上下文下推理（通过 `inference_rope_scaling=True` 启动）。
