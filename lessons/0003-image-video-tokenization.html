📸 图片与视频 Token 化

lesson 0003 阶段 3 — 多模态入门 · 视觉编码

你之前学过了文本 Tokenizer（BPE 把文字切成 token），知道了 LLM 只能吃 token 序列。 那问题来了——图片呢？视频呢？

毕竟 GPT-4o、通义千问都能看图，它们是怎么把一张照片或者一段视频"翻译"成 LLM 能理解的 token 的呢？

1. 核心问题：图片不是文字

回忆一下文本 Tokenizer 的工作方式：

"Hello world"  →  BPE 分词  →  ["Hello", " world"]  →  查词表拿 token ID  [15496, 1495]

文字天然是离散的——字母和词之间边界清楚。但图片呢？

一张 224×224 的彩色图片，本质上是一个 224×224×3 的浮点数矩阵（长×宽×RGB 三通道）。 是连续的，没有边界，没法直接查词表。

一句话总结问题：LLM 吃离散 token，图片是连续像素。要搭一座桥。

2. 解决办法：切格子（Patch Embedding）

2017 年 ViT（Vision Transformer）论文的标题就叫 An Image is Worth 16×16 Words——「一张图片值 16×16 个词」。

名字已经告诉你答案了：把图片切成一个个小方块，每个小方块就是一个"视觉单词"。

具体步骤（算例）

假设你拿到一张 224×224 的彩色照片：

步骤 1：选格子大小   →  16×16 像素（ViT 标准配置）
步骤 2：切分          →  224 ÷ 16 = 14
                         横向 14 个格子，纵向 14 个格子
                         总共 14×14 = 196 个格子
步骤 3：展平          →  每个格子 16×16×3 = 768 个数字（浮点数）
步骤 4：线性投影      →  768 个数字 × 一个大矩阵（W）→ 输出一个 768 维向量
                         这个向量就是「视觉 token」
步骤 5：加上位置编码  →  告诉模型每个格子原来在哪位置（左上角、右下角…）
步骤 6：送进 Transformer → 196 个视觉 token 像 196 个"词"一样被处理

每一步发生了什么，拿数据走一遍：

步骤输入形状做了什么输出形状
原图 224×224×3 — 224×224×3
切 patches 224×224×3 切成 14×14 个 16×16×3 小方块 196 × (768)
展平 196 个 16×16×3 每个方块拉直成 768 个数字 196 × 768
线性投影 196 × 768 乘权重矩阵 W: 768×768 196 × 768
+ 位置编码 196 × 768 加 RoPE 或可学习的位置向量 196 × 768
直觉类比：就像把一幅马赛克画数清楚——每个马赛克小方块是一个 "word"，马赛克排成 14×14 的格子，这些格子按顺序就是 LLM 能读的句子了。

一张图顶多少字？

一张 224×224 的图 ≈ 196 个视觉 token。而 196 个英文 token 大概能写一段 150 字的英文短文。 一张图和一段话，在模型眼里"长度差不多"。

3. 从图片到多模态 LLM

现在你有了视觉 token（196 个 768 维的向量）。可 LLM 的文本 token 也是向量啊——它们怎么共存？

两个问题需要解决：

  1. 维度对齐：视觉 token 是 768 维，LLM 的 embedding 可能也是 768 维——好巧！但如果不是呢？（比如 LLM 用 4096 维）
  2. 语义对齐：视觉 token 编码的是"颜色/边缘/纹理"这种底层信息，不是"猫/狗/车"这种语义。

主流解决方案有三种，我按复杂程度排序：

方案 A：MLP 投影层（LLaVA，2023）

最简单、最流行。在视觉编码器和 LLM 之间加一个 两层 MLP（一个简单的神经网络）：

视觉 token (196 × 768)
    ↓
MLP 投影层： 768 → 4096 → 4096
    ↓
对齐后的视觉 token (196 × 4096)
    ↓ 拼在文本 token 前面：
[img₁][img₂]...[img₁₉₆][系统指令][用户问题][模型回答]

关键：这个 MLP 就是整个多模态系统中唯一新增的可训练参数。视觉编码器和 LLM 都冻结不动，只训练这个小 MLP，让视觉 token 学会"翻译"成 LLM 能懂的语言。

算一笔账：LLaVA 用的投影层是 768→4096→4096，两层线性层。
参数 = 768×4096 + 4096×4096 ≈ 20M 参数。
相比一个 7B LLM（70亿参数）——这个投影层只有 LLM 的 0.03%。
所以 LLaVA 训练极其快，而且效果出奇好。

方案 B：交叉注意力（Flamingo / LLaMA 3.2 Vision）

LLM 每一层在计算完 self-attention 之后，额外加一个 cross-attention 层：

文本 token 处理流程（每层）：
[self-attention]  →  [cross-attention（读视觉 token）]  →  [FFN]

视觉 token 队列： [v₁][v₂][v₃]...[v₁₉₆] ← cross-attention 从这里读取

好处：视觉 token 不混入文本序列，每层独立决定"我这次想看哪些视觉信息"。

方案 C：Q-Former 压缩（BLIP-2）

引入可学习的查询向量（learned queries），通过 cross-attention 从视觉 token 中"提纯"出最相关的信息：

视觉 token × 256 ──────────────┐
                                ├→ cross-attention → 压缩后的 token × 32
可学习查询 × 32 ────────────────┘                                      ↓
                                                               送进 LLM

256 个视觉 token → 32 个"精华 token"。

4. 视频怎么办？Token 爆炸问题

如果一张 224×224 的图是 196 个 token，一段视频呢？

视频时长帧率总帧数总 token类比
10 秒1 fps10~2,0003 页书
10 秒30 fps300~58,800一本小册子
1 分钟30 fps1,800~352,800一本书
10 分钟30 fps18,000~3,528,000一整个书架

问题是显而易见的。目前主流模型上下文窗口是 128K~1M token，一部 10 分钟视频就干爆了。

解决方案：

  1. 帧采样：每秒只取 1-2 帧，丢掉中间帧。
  2. 帧压缩：每帧通过 Q-Former 从 196 个 token 压缩到 16-32 个。
  3. 时空编码器：用 3D 卷积一次编码多帧，直接输出更少的 token。
  4. 流式处理：不一次性加载全部视频，而是随看随丢旧帧（类似 KV Cache 的管理策略）。
现实情况：目前大多数"视频理解"模型其实主要看视频的文字描述和关键帧。真正的帧级视频理解还在研究中——计算量太大了。

6. 想一下

🤔 思考一：为什么 LLaVA 的 MLP 投影层能工作？

直觉上，"把视觉 token 过两层线性层"听起来太简单了。为什么它有效？

提示：CLIP 在预训练时已经把图片和文本对齐到了同一个 embedding 空间。视觉 token 和文本 token 在预训练后就已经在"同一个语言体系"里了，MLP 只是做最后的校准。

🤔 思考二：图片 token 数远多于文本 token 数有什么影响？

一张 224×224 的图 ≈ 200 个 token，也就是约 150 个英文单词。如果你上传一张图然后问"这张图里有什么？"——

视觉 token 占序列的 95%+。这意味着 Attention 矩阵的大部分计算都花在"看图片"上，而不是"理解问题+生成回答"上。
