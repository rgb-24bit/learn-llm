当一个新模型发布时，你肯定见过这样的表格：

典型模型发布宣传页

这些数字决定了模型采购、技术选型、甚至公司声誉。但很少有人能说清：MMLU 到底测什么？GSM8K 的 97% 是真的会算数吗？HumanEval 能代表写代码的能力吗？Arena Elo 评分又是怎么算出来的？

这节课就是你的基准全景图——把主流 LLM 测试按能力分类，每类讲清楚测什么、怎么测、有什么陷阱。

1. 基准的四个能力维度

LLM 的评估基准可以分成四大类，每类对应一种核心能力：

维度测什么代表基准格式
🧠 知识广度模型记住了多少事实MMLU, HellaSwag, TruthfulQA选择题
🔢 推理能力多步逻辑/数学推导GSM8K, MATH, GPQA开放回答
💻 代码能力能不能写出正确代码HumanEval, MBPP, SWE-bench函数生成
🗣️ 综合/人类偏好对话质量、有用性MT-Bench, Chatbot Arena模型/人类评分

注意这几个维度是正交的——一个模型可能知识渊博但数学很差，也可能代码写得飞起但一问常识就翻车。

2. 🧠 知识广度：MMLU

它是什么

MMLU（Massive Multitask Language Understanding） 由加州大学伯克利分校的 Hendrycks 等人在 2020 年提出。它包含 15,908 道四选一选择题，覆盖 57 个学科——从抽象代数到世界宗教，从临床医学到商法，从大学物理到中学心理学。

目标：测试模型在预训练阶段吸收了多少跨领域知识。

怎么测

  • 评分方式：逐题看模型选的选项（A/B/C/D）是否和正确答案一致
  • 默认设定：5-shot —— 每道题前先给 5 个例子（同一学科），让模型学会答题格式
  • 随机基线：25%（4 选 1 瞎蒙）
  • 两种评分方法：
    — 字母匹配：让模型生成文本，提取第一个 A/B/C/D 字母
    — Log-prob 打分：算出模型对 A/B/C/D 四个选项的对数概率，选最高的（更准确，因为模型可能不乱说字母）

具体数据

MMLU 核心数据
属性值
总题数15,908
学科数57
选项数4（A/B/C/D）
随机基线25%
人类专家~89%
GPT-3 (2020)~43%
前沿模型 (2026)89–93%
四大学科分组STEM · 人文 · 社科 · 其他
重要陷阱：选择题测的是「排除错误答案」，不是真正的知识。模型可能不认识病毒学，但认识「三个选项看起来不合理，选这个看起来合理的」。

MMLU-Pro — 更难版

因为 MMLU 已经被前沿模型打到了 90%+（接近天花板），2024 年有人推出了 MMLU-Pro：题目更难，选项从 4 个增加到 10 个，随机基线降到 10%。这样又拉开了模型差距。

3. 🔢 推理能力：GSM8K 与 MATH

GSM8K — 小学数学应用题

GSM8K（Grade School Math 8K） 是 OpenAI 2021 年发布的基准，包含 8,500 道小学数学应用题。每道题需要 2–8 步算术推导才能解出。

GSM8K 典型题目
题目：贝斯有 23 个苹果。她给了茉莉 7 个，又买了 12 个。
        她还有多少个苹果？

正确推导：23 - 7 = 16
          16 + 12 = 28
答案：28

GMS8K 的关键设计：题目必须逐步推理才能得到正确答案。模型如果只是「猜数字」很难蒙对。这推动了 Chain-of-Thought（思维链） 技术的普及——让模型先写推导过程再出答案。

重要陷阱——数据污染：GSM8K 已经被打到 99%。更严重的是，2023 年研究发现去掉和训练数据重叠的题目后，某些模型的分数从 95% 跌到 82%。说明模型记住了答案而不是真正会算。

MATH — 竞赛级数学

比 GSM8K 难得多——包含 12,500 道 AMC/AIME 级别的竞赛数学题（高中数学竞赛到大学预科）。2023 年时最好的模型也只做到 ~50%，现在前沿模型能到 85-90%。

基准难度题目数前沿分数饱和否？
GSM8K小学应用8,50099%✅ 饱和
MATH竞赛级12,500~85%⚠️ 接近中
AIME美国数学邀请赛30/年~70%❌ 未饱和

4. 💻 代码能力：HumanEval 与 SWE-bench

HumanEval — LeetCode 级

OpenAI 发布 HumanEval 时自己也说：这个基准的名字其实有自嘲意味——「用人来评估」永远不是完美的，但总比没有好」。它包含 164 道 Python 编程题，每道题是一个函数签名 + 文档字符串 + 一组单元测试。

模型的输出是完整的函数代码。评分标准：pass@k——生成 k 个候选，只要有一个通过所有单元测试就算过。

pass@k 的实际推导
常用的是 pass@1：只生成一次，看能不能跑通

也可以 pass@5：生成 5 个版本，只要 1 个能跑通就算对
→ 这给「随机多试几次」留了空间

通过率 = 1 - (C[n-k][t] / C[n][t])
其中 n=采样数, t=通过的次数, k=候选数

SWE-bench — 真正的"修 Bug"

HumanEval 的问题是太"LeetCode"了——写个独立函数和真正的软件工程差太远。SWE-bench（2024）翻了个大天：它要求模型面对一个真实的 GitHub 仓库 + 一个 Issue 描述，然后生成一个 pull request 级别的修复补丁。

维度HumanEvalSWE-bench
任务写一个独立函数修一个真实的 Bug
上下文一段 docstring整个代码仓库
题目数1642,294
前沿分数90%+40–80%
难度低高（至今没人满分）
代表什么算法能力软件工程能力

5. 🗣️ 综合偏好：Chatbot Arena

它是怎么运作的

由 LMSYS 组织运营的 Chatbot Arena 是目前最有影响力的综合对话质量排行榜。它的方法非常巧妙——让两个模型匿名对战，真人投票。

用户提出一个问题 ↓ 模型 A 和模型 B 各自回答（匿名，用户不知道谁是谁） ↓ 用户阅读两个答案，投票选"更好的那个" ↓ 系统用 Elo 算法更新两个模型的分数 ↓ （Elo 评分来自国际象棋——赢了强手加很多分，输给弱手扣很多分）

Elo 评分的直觉

假设两个模型对战了无数次，每个模型拥有了一个 Elo 分数。那么预测 A 战胜 B 的概率：

Elo 预测公式的直觉
E_A = 1 / (1 + 10^((Elo_B - Elo_A) / 400))

如果 Elo_A - Elo_B = 200：
  → E_A = 1 / (1 + 10^(-0.5)) = 1 / (1 + 0.316) = 0.76
  → 76% 的概率 A 赢

如果 Elo_A - Elo_B = 0（分数相等）：
  → E_A = 1 / (1 + 10^0) = 0.5
  → 50% 的概率 A 赢

实际更新：如果 A 赢了但预期胜率只有 40%，那 A 的 Elo 会加很多分。反之如果 A 赢了但预期胜率是 90%，那只加一点点。

为什么这个排行榜好：它不是测标准答案，而是测人类真的更喜欢哪个模型的对话。这比任何选择题都更接近真实使用体验。

6. 🧹 基准的四大陷阱

陷阱什么意思典型例子
1. 饱和 所有模型都接近满分，基准失去区分度 GSM8K 99%、HellaSwag 95%+、MMLU 90%+
2. 数据污染 测试题出现在训练数据中，分数来自记忆而非能力 GSM8K 去掉重叠题后某些模型跌 13%
3. 格式刷分 模型学会答题格式而非内容 MMLU 选择题格式让模型靠排除法得分
4. 基准≠生产 满分基准不意味着在真实场景好用 HumanEval 满分但做不对实际 issue
