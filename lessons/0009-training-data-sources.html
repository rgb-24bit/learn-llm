💾 训练数据的来源与优化

lesson 0009 阶段 5 — 评估与数据 · 数据篇

前面我们学了模型怎么训练、怎么评估。但有个问题一直没回答——训练数据到底从哪来的？

别忘了，GPT-3 训练了大约 3,000 亿 token，Llama 3 训练了 15 万亿 token。这些数据不可能靠人工写出来。它们来自一个巨大的、充满噪音的、需要精心处理的来源——整个互联网。

这节课就讲三件事：数据从哪来、怎么洗、怎么配。

1. 数据从哪来？

1.1 Common Crawl — LLM 数据的"石油"

Common Crawl 是一个非营利组织，从 2008 年开始定期抓取整个互联网的网页快照。每个月抓取一次，每次产生几十 TB 的原始数据。它占了几乎所有 LLM 训练数据的 80% 以上。

典型 LLM 的数据来源构成（Llama 3 级别）
来源占比特点
Common Crawl（网页）~80%最大量，但质量参差不齐
Wikipedia~3-5%高质量，但量小
书籍（Books）~5-10%高质量长篇文本，通常受版权限制
学术论文（arXiv/PubMed）~2-5%专业知识和推理
代码（GitHub）~5-10%代码能力的基础
其他（Reddit/社交媒体等）~2%对话风格/多样性
关键认知：光靠 Wikipedia 是远远不够的。整个 Wikipedia 英文版才约 40 亿词，是 Llama 3 训练数据的 1/3000。没有 Common Crawl，任何大模型都训不出来。

1.2 高质量来源为什么重要

虽然 Common Crawl 是主力，但不同来源提供不同的能力：

来源给模型带来了什么类比
Wikipedia结构化知识、事实准确性教科书——精炼、可信
书籍长篇推理、叙事连贯性长篇小说——学会"一直讲一件事"
学术论文逻辑论证、专业知识研究论文——学会严谨推理
代码逻辑思维、精确表达代码仓库——学会"说一不二"
社交媒体对话风格、口语表达朋友圈——学会日常聊天
注意：Common Crawl 里本来就包含 Wikipedia 和 GitHub 的内容（爬虫会抓取这些网站）。所以实际上的"数据来源"不是独立分类，而是从 Common Crawl 这个大海里筛选不同区域的结果。

2. 数据清洗流水线

从 Common Crawl 抓下来的原始数据长什么样？大概像这样：

🛒 Buy Cheap Viagra Online!!! - Best Prices
...

Welcome to my blog. Today...

（后面跟了一段从 Wikipedia 抄的文本）

访客统计: 12345 | 友情链接...

这需要经过一系列清洗步骤：

Step 1: HTML 提取

把 HTML 标签去掉，只留下可视文本。工具如 jusText 或 trafilatura 还能识别导航栏、广告区域，只保留正文内容。

Step 2: 语言识别

用 FastText 或 langid 判断文本是什么语言。只需要英文的话，就把中文、俄文、阿拉伯文等全部扔掉。

大坑：很多英文网页里有大量非英文内容（比如技术博客里嵌入的中文评论）。语言分类器可能因为标题是英文就判定整页是英文，结果混入了大量噪音。

Step 3: 启发式过滤（Heuristic Filtering）

用简单的规则快速筛掉明显的"垃圾"——这是成本最低的过滤方式。

过滤规则去掉什么
长度过滤少于 50 个字符的段落（标题页、404）
符号比例标点符号占比过高的文本（数据表格、乱码）
重复行比例同一行出现 30 次以上的（模板文本）
停用词密度英文停用词（the/a/an/is）占比太低 → 可能不是自然语言
URL 占比文本里链接太多的（导航页、聚合页）
整句比例句号结尾的句子不足 50%（碎片文本）
内容类型去掉"赌博/色情/黑五/购买"等关键词

FineWeb（Hugging Face 2024 年发布的 15 万亿 token 数据集）从 50 多种候选过滤规则中只保留了 7 种最有效的——发现大多数规则对模型效果没有帮助。

Step 4: 去重（Deduplication）

互联网上充满了重复和近似重复的内容——同一篇文章被不同网站转载、同一段代码在 Stack Overflow 和 GitHub 各出现一次。去重是提升数据质量最关键的一步。

去重分三个层次：

去重的三个层次
层次方法效果计算成本
1. 精确去重 计算每篇文档的 hash，只保留一份 去掉完全相同的副本 低（O(n)）
2. 模糊去重 MinHash + LSH——把文档转成指纹，找相似度高的 去掉几乎一样的（改动几个词的抄袭文章） 中（O(n log n)）
3. 语义去重 用 embedding 找语义相似的文档 去掉同一事件不同报道（内容实质重复） 高（需要跑模型）

模糊去重（MinHash） 是最常用的方法。它怎么工作？

文档 → 拆成 n-gram（连续 n 个词，比如 5-gram） → 每个 n-gram 算一个 hash → 用 MinHash 取最小值 → 得到一个"文档指纹" → LSH（局部敏感哈希）把相似指纹分到同一个"桶"里 → 同一桶里的文档两两比较相似度 → 相似度 > 阈值（如 0.85）→ 只保留一个

SlimPajama-DC 研究发现：全局去重（跨所有来源去重）比局部去重（各自去重）效果好得多——因为同一个知识可能同时出现在 Wikipedia 和 Common Crawl 里，各自去重删不掉交叉重复。

Step 5: 基于模型的过滤（Model-based Filtering）

启发式规则能筛掉明显的垃圾，但分不清"普通文章"和"高质量文章"。2024 年以来，最先进的清洗方法是用一个小模型来判断文档质量。

FineWeb-Edu 的过滤方法（Hugging Face, 2024）
1. 用 Llama 3 70B 给 50 万篇文档标注"教育质量"
   → 每个文档打 1-5 分（完全没用 → 高度教育性）

2. 用这些标注训练一个小分类器（基于 fastText）
   → 只有 5 层神经网络，极轻量

3. 把整个 FineWeb（15T token）过一遍分类器
   → 保留分数 ≥ 3 的文档（约占总量 8~10%）

4. 得到 FineWeb-Edu：1.3T token 的高质量教育数据

结果：用 FineWeb-Edu 训练的模型，
      MMLU 分数比用原始 FineWeb 高 5-10 个百分点！

这就是"用大模型来教小模型怎么挑数据"——Llama 3 70B 当评委，fastText 当执行者。成本低、效果好。

代价：这种过滤会筛掉 90% 的数据。FineWeb-Edu 只有 1.3T token，只有原始 FineWeb 的 8.7%。对于超大模型（如 Llama 3 405B 需要 15T token），光靠高质数据不够——你只能反复重复看同一份数据，效果反而变差。

Full Pipeline 一览

Common Crawl（~50TB/月 原始 HTML） │ ▼ Step 1: HTML → 纯文本（jusText / trafilatura） │ ▼ Step 2: 语言识别（FastText） │ ▼ Step 3: 启发式过滤（长度/符号/重复率） │ ▼ Step 4: 去重（精确 + MinHash 模糊去重） │ ▼ Step 5: 模型质量过滤（FineWeb-Edu 分类器） │ ▼ 最终训练集（万亿 token 级别）

3. 数据混合（Data Mixing）

数据洗完了，每种来源都有了一大堆。接下来就是按什么比例混在一起。

SlimPajama-DC 的实验发现

Cerebras 和 MBZUAI 的研究人员在 2023 年发表了 SlimPajama-DC 研究，系统性地对比了不同数据混合策略。他们的核心发现：

  1. 代码不能少——去掉代码后，模型在推理任务上明显变差。代码训练了模型的"逻辑思维"
  2. 书籍比例要多——书籍提供长篇推理能力，对知识型基准（MMLU）提升明显
  3. Wikipedia 量小但值钱——虽然占比小，但增量收益极高
  4. 多样性很关键——全局去重后，来源多样性对模型泛化很重要

典型数据配比

数据源Llama 1 配比Llama 2Common Corpus
Common Crawl67%~89%~75%
Wikipedia4.5%~2.5%~5%
书籍15%~4%~10%
代码4.5%~3%~7%
学术论文4.5%—~3%
其他4.5%~1.5%—
注意：这些配比是最终训练集的占比。但因为去重和过滤会不均匀地淘汰不同来源的内容，原始输入比例会完全不一样。比如 Common Crawl 原始占比是 99%+，但经过过滤后可能只剩 80%。

Chinchilla 法则的数据面

你还记得阶段 3 学过的 Scaling Law 吗？Chinchilla 说"20 token 比 1 参数"。但这里有个隐藏前提——用的数据质量足够高。

如果你的数据全是垃圾，那 100 个 token 也比不上 1 个参数。反过来，FineWeb-Edu 这类超高质量数据可以用更少的 token 达到同样的 MMLU 分数——也就是说数据质量可以"节约"数据量。

4. 数据污染与去污染

数据污染（Data Contamination）是指：测试集的内容不小心混进了训练数据。

因为 Common Crawl 是抓取整个互联网的，而 MMLU/GSM8K/HumanEval 的测试集又都是公开在网上的——你不刻意排除它们，它们就自动存在于训练数据里。

后果：模型在测试集上的得分高得虚假，因为它已经"见过答案"了。

去污染的典型做法

1. 准备"污染检测集" → 收集所有要评估的 benchmark 的测试数据 2. 对每篇训练文档，检测是否和测试数据重叠 → 精确匹配：直接 substring 搜索 → 近似匹配：n-gram 重叠率 > 阈值 3. 发现重叠 → 删除整篇文档 4. 难题：变体污染（Paraphrased Contamination） → 测试题被改写后出现在训练数据中 → 无法通过精确/近似匹配检测 → 仍然会拉升分数

动态基准——数据污染的终极解决方案

既然静态测试集注定会被污染，行业正在转向动态基准（如 LiveBench）：每月更新全新题目，确保模型没机会提前"见过"。

5. SFT/LoRA 的数据特点

上面讲的主要是 Pretrain 数据。对于 SFT 数据，情况不太一样：

维度Pretrain 数据SFT 数据
数据量万亿 token几千~百万条对话
来源Common Crawl 自动抓取人工 + 模型生成 + 混合
清洗重点去噪、去重去错（回答质量审查）
多样性来源多样性场景多样性
格式纯文本对话格式（role/content）

SFT 数据的质量 > 数量是一个非常强的结论。InstructGPT 只用 13K 条高质量人工标注就训练出了远超 GPT-3 的对话能力。2023 年周伯峰等人的论文也发现：用 1,000 条精心挑选的数据做 SFT，效果可以比 50,000 条未筛选的数据好。

6. 关键启示与应用

关于训练数据的三句话
  1. Common Crawl 是石油，但需要精炼——原始数据 90% 是垃圾，不到 10% 是真正有用的
  2. 去重 > 过滤——去掉重复数据比去掉低质数据的收益更大（因为重复数据让模型过拟合到常见模式）
  3. "用大模型挑数据"是最佳实践——用 Llama 3/FineWeb-Edu 分类器的方法，比启发式规则有效得多
