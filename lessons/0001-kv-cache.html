🗂️ KV Cache

lesson 0001 阶段 3 — GPT 架构 · 推理优化

KV Cache 是让 LLM 生成文本 不重新计算历史 K/V 的核心优化。没有它，生成 2000 个 token 的 Attention 计算量是 O(n²)，有了它降到 O(n)。

1. 问题：Train vs Inference 的差异

训练时：一次输入整句话，所有位置并行计算 Attention，效率极高。

推理生成时：一个字一个字往外蹦，序列每步 +1。

训练（并行）：  ["Hello", "how", "are", "you"] → 4 个位置一起算
推理（串行）：  输入 "Hello"    → 输出 "how"
               输入 "Hello how" → 输出 "are"
               输入 "Hello how are" → 输出 "you"

最笨的做法：每步重新计算所有历史位置的 QKV。第 n 步时，你算第 1 步的 K 算到第 n 步的 K，但第 1 步到第 n-1 步的 K 跟上次结果一模一样——白算。

复杂度分析

步骤无 Cache有 Cache
第 1 步算 K₁, V₁算 K₁, V₁ → 存 cache
第 2 步算 K₁, V₁ + K₂, V₂只算 K₂, V₂ → 追加到 cache
第 3 步算 K₁, V₁ + K₂, V₂ + K₃, V₃只算 K₃, V₃ → 追加到 cache
第 n 步n 次 KV 计算1 次 KV 计算
累计 KV 层计算量对比： 无 Cache: 1 + 2 + 3 + ... + n = n(n+1)/2 ≈ O(n²) 有 Cache: 1 + 1 + 1 + ... + 1 = n ≈ O(n) n=2000: 无 Cache ≈ 2,001,000 次 KV 计算 有 Cache ≈ 2,000 次 KV 计算
核心洞察：Attention 的 Q×K^T 点数计算量没变（每步还是算 n 个点积），但 K 和 V 本身的生成从 O(n²) 降到 O(n)。实际推理中 KV 生成是瓶颈，因为涉及整个 Linear 层的矩阵乘。

2. 具体怎么运作

2.1 每步存什么

每个 Attention 头的 K 和 V 都要存。Q 不存——每次生成新 token 需要全新的 Q。

模型：hidden_size=768, num_heads=8, head_dim=96

每生成 1 个 token（1 层）：
    新 K  = 8 头 × 96 = 768 个数字
    新 V  = 8 头 × 96 = 768 个数字
    每层新增 = 1,536 个 float

8 层（minimind）：
    每步新增 = 8 × 1,536 = 12,288 个 float ≈ 48KB (float32)
    生成 2000 词 → 约 96MB

2.2 细粒度步骤

  1. Prefill 阶段：第一次输入整条 prompt（比如用户的问题 "中国的首都是"），一次算完所有位置的 K 和 V，存入 cache
  2. 生成第 1 个 token：只用新来的 Q 对 cache 里的所有 K 做 Attention → 产出 "北京"
  3. 追加 cache：算 "北京" 这个位置的 K_new, V_new → 追加到 cache
  4. 生成第 2 个 token：用新的 Q 对 cache（原 prompt + "北京"）做 Attention → 产出 "是"
  5. 重复直到遇到结束符或达到 max_tokens

2.3 Prefill vs Decode

Prefill（首个 token）Decode（后续 token）
输入整条 prompt（如 200 个 token）1 个新 token
KV Cache空 → 填满所有 prompt 位置已有 → 每次追加 1 个
计算高吞吐（批量并行）低吞吐（串行 1 个）
瓶颈算力内存带宽（读 cache）
实际推理中 Decode 是更慢的阶段——虽然只算 1 个新 token，但 Attention 要把新 Q 跟 cache 里所有 K 做点积。序列长了之后，读 cache 的内存带宽成为瓶颈。这就是为什么 Flash Attention 这类优化主要加速 Decode 阶段。

3. 内存消耗——为什么上下文窗口有限

典型模型 KV Cache 内存估算
模型层数dim头数每 token/层4K 上下文8K 上下文
minimind876886KB24MB48MB
GPT-2 124M127681212KB48MB96MB
LLaMA 7B324096321MB4GB8GB
LLaMA 70B808192645MB20GB40GB

这就是为什么：

  • 70B 模型跑 32K 上下文需要 160GB+ 显存（仅 KV Cache 就超出单张 A100 80GB）
  • Multi-Query Attention (MQA) 和 Grouped-Query Attention (GQA) 被发明——多个 Q 头共享同一组 K/V，cache 降到 1/8
  • 这是推理优化（vLLM、TensorRT-LLM）的核心战场——管理 KV Cache 内存

4. 与已学知识的连接

已学概念跟 KV Cache 的关系
RoPE 位置编码旋转在入 Cache 前完成，Cache 存的已经是旋转后的 K
Causal AttentionCausal mask 在生成时自然满足——新 token 永远只 attend 到已有的历史
Multi-Head Attention每个头独立维护自己的 KV Cache
GPT Block 组装每层 Block 都有自己独立的 KV Cache，不跨层共享
QKV 计算只有 Q 每步全重新算；K 和 V 只算新来的

6. 关键点总结

特性值
什么被 Cache每层每个 Attention 头的 K 和 V
什么不 CacheQ（用完就扔）
省什么避免重复计算历史 K/V（O(n²)→O(n)）
代价显存占用与序列长度成正比
在哪用Decoder-only 自回归生成（GPT, LLaMA, Qwen）
Cache 生命周期一次对话/生成任务的整个周期
跨层共享？不，每层独立

一句话：每次生成新词时不重新计算历史位置的 K 和 V，存起来复用。用内存换速度。
